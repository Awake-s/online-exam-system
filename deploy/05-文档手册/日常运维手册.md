## 1. 日常开发改代码 → 上线

### 🎯 完整流程（最常用的运维动作）

```
[本地] 改代码 → 本地测试通过 → 提交 Git
[本地] .\deploy\02-更新上线\local-build.ps1 -Target backend       ← PowerShell，一键构建+上传
[服务器] ssh + sudo bash /opt/exam-system/update.sh backend   ← 应用新版本
```

### 1.1 首次使用（只需做一次）

上传 `update.sh` 和 `cleanup-test-data.sh` 到服务器，并初始化 staging 目录：

**本地 PowerShell**：

```powershell
# 上传运维脚本
scp "d:\Java Projects\在线考试系统\deploy\02-更新上线\update.sh" ubuntu@124.222.21.219:~/deploy/update.sh
scp "d:\Java Projects\在线考试系统\deploy\03-日常维护\cleanup-test-data.sh" ubuntu@124.222.21.219:~/deploy/cleanup-test-data.sh

# 创建服务器 staging 目录
.\deploy\02-更新上线\local-build.ps1 -SetupStaging
```

**服务器 SSH**：

```bash
# 部署运维脚本到 /opt/exam-system/
sudo cp ~/deploy/update.sh /opt/exam-system/update.sh
sudo cp ~/deploy/cleanup-test-data.sh /opt/exam-system/cleanup-test-data.sh
sudo chmod +x /opt/exam-system/update.sh /opt/exam-system/cleanup-test-data.sh
sudo chown root:root /opt/exam-system/update.sh /opt/exam-system/cleanup-test-data.sh
ls -la /opt/exam-system/*.sh
```

### 1.2 只改了**后端**代码（Java）

**本地 PowerShell**（在项目根目录下）：

```powershell
.\deploy\02-更新上线\local-build.ps1 -Target backend
```

脚本会自动：
1. `mvn clean package -DskipTests`（约 2-3 分钟）
2. `scp` 上传 JAR 到服务器 `/home/ubuntu/staging/`
3. 输出"下一步"提示

**服务器 SSH**：

```bash
sudo bash /opt/exam-system/update.sh backend
```

脚本会自动：
1. 备份当前 JAR 到 `/opt/exam-system/backups/releases/jar/`
2. 替换 JAR 为新版本
3. `systemctl restart exam-system`
4. 等待 `/actuator/health` 返回 UP
5. **如果健康检查失败 → 自动回滚到上一版本**

### 1.3 只改了**前端**代码（Vue）

**本地 PowerShell**：

```powershell
.\deploy\02-更新上线\local-build.ps1 -Target frontend
```

**服务器 SSH**：

```bash
sudo bash /opt/exam-system/update.sh frontend
```

前端更新**不需要重启后端**，Nginx reload 即可生效。用户浏览器刷新就能看到新版本。

### 1.4 前后端都改了

**本地**：

```powershell
.\deploy\02-更新上线\local-build.ps1 -Target all
```

**服务器**：

```bash
sudo bash /opt/exam-system/update.sh all
```

### 1.5 只打包，不上传（调试用）

```powershell
.\deploy\02-更新上线\local-build.ps1 -Target backend -DryRun
```

### 1.6 已经手动打包过了，只想上传

```powershell
.\deploy\02-更新上线\local-build.ps1 -Target backend -SkipBuild
```

## 2. 清理测试数据

### 🎯 使用场景

上线前用管理员/教师/学生账号测试过，数据库里积累了假考试、假聊天、假通知。**正式上线给真实用户前必须清理一次**。

### 2.1 三档清理级别

| 级别 | 清理范围 | 使用场景 |
|---|---|---|
| `--basic` | 通知 + 聊天消息/会话 | 只是测了下登录，不想删考试 |
| `--full` | basic + 考试 + 考试记录 + 答题 | **上线前推荐**，试卷和题库保留 |
| `--reset` | full + 试卷 + 试卷模板 | 彻底回到"只有题库"状态 |

**永远不清理**：用户、角色、班级、专业、科目、教师-班级/科目关联、**题库**。

### 2.2 先预览（不会真删）

```bash
sudo bash /opt/exam-system/cleanup-test-data.sh --full --dry-run
```

输出会显示每个表当前有多少行会被删。

### 2.3 正式清理（推荐）

```bash
sudo bash /opt/exam-system/cleanup-test-data.sh --full
```

脚本会：
1. 显示将清理的表和行数
2. 要求输入 `yes` 确认
3. 自动全库备份到 `/opt/exam-system/backups/cleanup/`
4. 按外键顺序 TRUNCATE
5. 清理 Redis 里 `exam:*` 缓存 key
6. 验证清理结果

### 2.4 自动化清理（跳过确认）

```bash
sudo bash /opt/exam-system/cleanup-test-data.sh --full --yes
```

### 2.5 万一清错了，从备份恢复

```bash
# 找到最近的备份
ls -t /opt/exam-system/backups/cleanup/

# 恢复（需要 MySQL root 密码）
sudo gunzip < /opt/exam-system/backups/cleanup/before_cleanup_full_20260423_150000.sql.gz \
    | sudo mysql -u root -p online_exam_system
```

## 3. 回滚到上一版本

### 🎯 使用场景

上线新版本后发现有严重 bug，想立即回到上一个正常版本。

### 3.1 一键回滚（前端 + 后端）

```bash
sudo bash /opt/exam-system/update.sh rollback
```

脚本会自动找到 `/opt/exam-system/backups/releases/` 里**最近的**一个 JAR 和 dist 备份，恢复并重启。

### 3.2 手动回滚到指定版本

```bash
# 1. 看有哪些备份
ls -la /opt/exam-system/backups/releases/jar/
ls -la /opt/exam-system/backups/releases/frontend/

# 2. 手动替换（示例：回滚到 20260422_232900 的版本）
sudo cp /opt/exam-system/backups/releases/jar/exam-system-1.0.0.jar.20260422_232900.bak \
        /opt/exam-system/exam-system-1.0.0.jar
sudo chown exam:exam /opt/exam-system/exam-system-1.0.0.jar
sudo systemctl restart exam-system

# 3. 健康检查
curl -s http://127.0.0.1:8081/actuator/health | jq
```

**注意**：update.sh 默认只保留**最近 5 个** JAR 备份和 **最近 3 个**前端备份。如需更长的版本留存，手动复制到 `/home/ubuntu/` 等位置。

## 4. 服务状态查看

### 4.1 一键健康体检（每周跑一次）

```bash
echo "=== 服务状态 ===" && sudo systemctl is-active exam-system nginx mysql redis-server
echo "=== 磁盘 ===" && df -h / | head -2
echo "=== 内存 ===" && free -h
echo "=== 备份 ===" && ls /opt/exam-system/backups/mysql/ | tail -5
echo "=== 近 5 次备份任务 ===" && sudo tail -5 /opt/exam-system/logs/backup.log
echo "=== 后端最近异常 ===" && sudo journalctl -u exam-system --since '1 day ago' -p err --no-pager | tail -10
```

### 4.2 具体状态命令

```bash
# 后端服务
sudo systemctl status exam-system
sudo journalctl -u exam-system -n 100 --no-pager

# 实时看日志
sudo tail -f /opt/exam-system/logs/stdout.log

# Nginx
sudo systemctl status nginx
sudo nginx -t

# MySQL
sudo systemctl status mysql
mysql -u exam_app -p online_exam_system -e "SELECT COUNT(*) FROM sys_user;"

# Redis
redis-cli -a "$(sudo grep '^REDIS_PASSWORD=' /etc/exam-system/secrets.env | cut -d= -f2-)" \
          --no-auth-warning PING

# 端口监听
sudo ss -tlnp | grep -E '80|443|8081|3306|6379'
```

### 4.3 健康检查接口

```bash
# 完整体检
curl -s http://127.0.0.1:8081/actuator/health | jq

# 单项
curl -s http://127.0.0.1:8081/actuator/health | jq '.components.db.status'
curl -s http://127.0.0.1:8081/actuator/health | jq '.components.redis.status'
curl -s http://127.0.0.1:8081/actuator/health | jq '.components.diskSpace.status'
```

## 5. 排查业务 bug

### 5.1 场景：用户反馈"某个页面白屏/报错"

**Step 1 - 前端控制台**：

让用户在有问题的页面按 **F12** → Console / Network 截图给你。

**Step 2 - 找对应的后端请求**：

```bash
# 看 Nginx 访问日志，找到对应的 API 请求
sudo tail -50 /var/log/nginx/exam-access.log | grep -E 'POST|PUT|DELETE'

# 过滤错误响应（4xx/5xx）
sudo grep -E ' (4|5)[0-9]{2} ' /var/log/nginx/exam-access.log | tail -20
```

**Step 3 - 看后端日志**：

```bash
# 实时跟踪
sudo tail -f /opt/exam-system/logs/stdout.log

# 找最近的 ERROR 堆栈
sudo grep -A 20 'ERROR' /opt/exam-system/logs/stdout.log | tail -50

# journalctl 按时间查
sudo journalctl -u exam-system --since '10 minutes ago' --no-pager
```

**Step 4 - 如果需要重现**：

```bash
# 看具体的 API 请求参数
sudo tail -f /var/log/nginx/exam-access.log &
# 然后让用户复现问题，看 URL、状态码、耗时
```

### 5.2 场景：某个功能慢

```bash
# Nginx 访问日志里带有 $request_time（最后一列，秒）
sudo awk '$NF > 1.0' /var/log/nginx/exam-access.log | tail -20

# MySQL 慢查询（deploy.sh 已配置记录 > 1s 的查询）
sudo tail -50 /var/log/mysql/mysql-slow.log
```

### 5.3 场景：修改代码想热调试

**不推荐**直接在服务器上改源码。正确流程：

1. 本地复现问题（用生产数据库的**备份**在本地还原，避免影响线上）
2. 本地修复并测试
3. 通过 `local-build.ps1` + `update.sh` 部署

### 5.4 从生产拉一份数据到本地调试

**服务器**（拿一份最新备份）：

```bash
ls -t /opt/exam-system/backups/mysql/ | head -1
# 假设是 db_20260423_030000.sql.gz
```

**本地 PowerShell**：

```powershell
# 拉到本地
scp ubuntu@124.222.21.219:/opt/exam-system/backups/mysql/db_20260423_030000.sql.gz D:\temp\

# 在本地 MySQL 导入一个副本库（别覆盖本地开发库！）
# 先 gunzip
7z x D:\temp\db_20260423_030000.sql.gz -oD:\temp\

# 在 MySQL 里
# CREATE DATABASE online_exam_system_prod_copy;
# mysql -u root -p online_exam_system_prod_copy < D:\temp\db_20260423_030000.sql
```

本地复现完问题就删掉这个副本库，**不要留太久（含真实用户数据）**。

## 6. 备份与恢复

### 6.1 备份策略（已自动化）

- **每天 03:00** 自动备份（crontab 配置）
- **保留 30 天**，超时自动删
- **位置**：`/opt/exam-system/backups/{mysql,redis,uploads}/`

### 6.2 手动立即备份一次

```bash
sudo bash /opt/exam-system/backup.sh
```

### 6.3 查看备份状态

```bash
# 看最近的备份
sudo ls -laht /opt/exam-system/backups/mysql/ | head -10

# 查备份日志
sudo tail -30 /opt/exam-system/logs/backup.log
```

### 6.4 从备份恢复 MySQL

```bash
# 1. 停止后端避免数据冲突
sudo systemctl stop exam-system

# 2. 恢复数据库（会覆盖当前！）
sudo gunzip < /opt/exam-system/backups/mysql/db_20260423_030000.sql.gz \
    | sudo mysql -u root -p online_exam_system

# 3. 重启
sudo systemctl start exam-system

# 4. 验证
curl -s http://127.0.0.1:8081/actuator/health
```

### 6.5 异地备份（可选加强）

如果担心腾讯云机房出事，建议定期下载备份到本地：

```powershell
# 本地 PowerShell，每周跑一次
$date = Get-Date -Format 'yyyyMMdd'
New-Item -Type Directory -Force "D:\exam-system-backups\$date" | Out-Null
scp -r ubuntu@124.222.21.219:/opt/exam-system/backups/mysql "D:\exam-system-backups\$date\"
```

## 7. 常见问题速查

### 7.1 `sudo systemctl restart exam-system` 后一直 failed

```bash
# 看错误原因
sudo journalctl -u exam-system -n 50 --no-pager
sudo tail -50 /opt/exam-system/logs/stdout.log

# 常见原因：
# 1. JAR 损坏 → 重新上传 + update.sh
# 2. 端口 8081 被占 → sudo ss -tlnp | grep 8081
# 3. MySQL/Redis 不通 → 检查密码、重启这两个服务
# 4. secrets.env 有错 → sudo cat /etc/exam-system/secrets.env
```

### 7.2 `local-build.ps1` 构建失败

| 错误 | 原因 | 解决 |
|---|---|---|
| `mvn 未安装` | 本地没装 Maven | 装 Maven 3.6+ 并加入 PATH |
| `npm 未安装` | 本地没装 Node | 装 Node 18+ |
| `BUILD FAILURE` | 代码编译错 | 看具体错误修代码 |
| `scp: open local` | 通配符/路径问题 | 在项目根目录跑脚本 |
| `Connection refused` | 服务器不通 | ping 服务器 / 检查 22 端口 |

### 7.3 `update.sh backend` 报健康检查失败

看具体日志：

```bash
sudo tail -100 /opt/exam-system/logs/stdout.log
```

如果自动回滚成功，系统已经回到上一个正常版本。排查完新 JAR 的 bug 后再重试。

### 7.4 服务器磁盘满

```bash
# 看哪里占用大
sudo du -h --max-depth=2 / 2>/dev/null | sort -hr | head -20

# 常见占用大头：
# - /opt/exam-system/backups/（30 天备份）
# - /var/log/（Nginx + MySQL 日志）
# - /opt/exam-system/logs/（Spring Boot 日志，看 logback 配置轮转）

# 快速清理：删 30 天前的 Nginx 日志
sudo find /var/log/nginx/ -name '*.gz' -mtime +30 -delete

# 缩短备份保留期（从 30 天改为 14 天）
sudo sed -i 's/RETENTION_DAYS=30/RETENTION_DAYS=14/' /opt/exam-system/backup.sh
```

### 7.5 Nginx 502 Bad Gateway

说明 Nginx 转发到后端失败（后端挂了 or 响应超时）。

```bash
# 1. 检查后端
sudo systemctl status exam-system
curl -v http://127.0.0.1:8081/actuator/health

# 2. 看 Nginx 错误日志
sudo tail -30 /var/log/nginx/exam-error.log

# 3. 如果后端真挂了
sudo systemctl restart exam-system
```

### 7.6 忘了 MySQL/Redis 密码

```bash
# 查看密码（只在本地打开 secrets.env 文件，不要在共享屏幕时查！）
sudo cat /etc/exam-system/secrets.env
```

**注意**：这是**单点故障**。本地应该保留一份凭据副本（部署时已提醒保存）。

### 7.7 想加新的环境变量到后端

```bash
# 编辑 secrets.env
sudo vim /etc/exam-system/secrets.env
# 加一行：MY_NEW_VAR=value

# 重启后端生效
sudo systemctl restart exam-system
```

代码里用 `@Value("${my.new.var}")` 或 `${MY_NEW_VAR}` 读取。

## 🆘 真的搞不定时的求救顺序

1. **查文档**：本手册 + `部署上线指南.md` + `上线实战手册.md`
2. **查日志**：`journalctl` / `/opt/exam-system/logs/` / `/var/log/nginx/`
3. **拉备份到本地复现**：用生产数据的备份在本地重现问题
4. **回滚**：`sudo bash /opt/exam-system/update.sh rollback`
5. **完全故障时**：腾讯云控制台 → 关机 → 从最近的**整机快照**恢复
