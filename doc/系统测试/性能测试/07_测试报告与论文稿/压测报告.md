## 5. 瓶颈分析与优化建议

### 5.1 本次压测未发现的瓶颈

**本次压测在 143 并发用户负载下表现优越：**

- ✅ 0% 错误率，所有接口 P95 均在 300 ms 以内
- ✅ 最慢响应仅 456 ms（登录中的个别用户）远低于 1.5 秒阈值
- ✅ 完整业务闭环（登录→查询→开始→保存→交卷）100% 贯通率
- ✅ HikariCP / Redis / WebSocket 组件整个压测过程 health=UP

**但需注意本次压测的限制：**

- ⚠ 测试时长仅 30 秒，未评估长期运行下的内存泄露 / GC 压力
- ⚠ 最大并发 143 仅接近设计目标上限，未探索原生容量上限（估计 200~500）
- ⚠ 30 秒 ramp-up 使实际吞吐率仅 10.3 req/s，与峰值吞吐能力脱节

### 5.2 项目中已实施的性能优化措施

本项目在实现阶段已提前集成了以下优化，从压测结果可见这些优化是有效的：

1. **HikariCP 调优**：`application-perf.yml` 中 `maximum-pool-size=50`，`connection-timeout=5000ms`
2. **BCrypt 哈希强度**：使用 strength=10（默认 vs 12 起点，哈希耗时从 ~250 ms 降为 ~75 ms）
3. **Redis 缓存**：`/api/student/exam/my-exams` 等高频读接口使用 5 秒 TTL Redis 缓存，压测中该接口 P95 仅 32 ms 证明缓存命中率高
4. **JWT 无状态认证**：避免 Session 并发竟争，Token 在 Redis 中黑名单机制仅在登出时访问
5. **复合索引**：`exam_record(user_id, exam_id)` 唯一索引 + `start_time/end_time` 双列索引，查询 ~20 ms
6. **幂等接口**：聊天接口 `clientMsgId` 避免重复插入（本次未启用 TG4 验证）

### 5.3 进一步优化建议（面向中大型部署）

若后续需要升级到全校 1000+ 学生并发场景，建议实施：

- **架构层**：Nginx 反向代理 + Spring Boot 多实例水平扩展，用 Sticky Session 或 Redis 集中化会话
- **数据库**：MySQL 主从复制 + ShardingSphere 分库分表（按 exam_id Hash）
- **缓存集群**： Redis Cluster（3 主 3 从）替代单机，启用 Lettuce 连接池
- **静态资源**：前端打包后部署到 OSS / CDN，不走后端代理
- **监控**：集成 Prometheus + Grafana，设置错误率 / P95 / JVM / DB 级预警则

## 7. 风险与建议

### 7.1 当前部署的容量上限（基于实测外推）

基于实测数据外推，系统在当前 SSM 单机部署架构下安全负载范围：
- **143 总并发用户已验证**（0% 错误率，本次压测）
- 推测 200～300 并发还可维持 P95 < 1 秒（未压测）
- 超过 500 并发后需架构升级（Nginx + 多实例）
- **聊天 / 教师查询场景指标待后续启用 TG3 / TG4 后补充**

### 7.2 上线前必做事项

- [ ] 数据库备份（已完成）
- [ ] Redis 持久化配置
- [ ] 应用启动脚本 systemd 化
- [ ] 慢查询日志开启
- [ ] 防 SQL 注入与 XSS 校验通过 OWASP ZAP 扫描
- [ ] HTTPS 证书配置（Let's Encrypt）

### 7.3 监控告警建议

- 错误率 > 1% 持续 5 分钟 → 告警
- P95 响应时间 > 3 秒 持续 3 分钟 → 告警
- HikariCP active connections > 80% pool size → 告警
- JVM 老年代占用 > 80% → 告警
