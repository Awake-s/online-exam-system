# 测试设计技术说明 (Test Design Techniques Specification)

> **标准依据**: ISO/IEC/IEEE 29119-4:2021 *Test techniques* + ISTQB Foundation Level Syllabus v4.0  
> **适用对象**: 在线考试系统接口测试 v1.6（145 用例 / 216 断言 / 100% 通过）  
> **作者**: 陶展 | **修订**: 2026-04-30

---

## 一、设计技术概览

ISO/IEC/IEEE 29119-4:2021 与 ISTQB 将测试设计技术分为三大类：

| 类别 | 子技术 | 本测试集应用 |
|---|---|---|
| **黑盒规约技术**（Specification-based） | 等价类划分、边界值分析、决策表、状态迁移、用例测试 | ✅ 5 项中 4 项 |
| **白盒结构技术**（Structure-based） | 语句覆盖、分支覆盖、路径覆盖 | N/A（接口测试本质黑盒）|
| **经验技术**（Experience-based） | 错误推测、探索性测试 | ✅ 错误推测大量使用 |

---

## 二、等价类划分 (Equivalence Partitioning)

### 2.1 原理

ISTQB 定义：**将输入域划分为若干等价类**（同一类中所有元素被假设具有相同的处理行为），从每类挑选 1 个代表值即可。

### 2.2 本测试集应用

#### 案例 1：用户登录角色等价类

| 等价类 | 代表值 | 用例数 | 预期 |
|---|---|---|---|
| 有效角色 ADMIN | `it_admin / Admin@2026` | 1 | HTTP 200 / code=200 |
| 有效角色 TEACHER | `it_teacher / Teacher@2026` | 1 | HTTP 200 / code=200 |
| 有效角色 STUDENT | `it_student / Student@2026` | 1 | HTTP 200 / code=200 |
| 无效用户名 | `nonexistent_test_user_xyz` | 1 | HTTP 200 / code=409 |
| 空用户名 | `""` | 1 | HTTP 200 或 400 |

**3 个有效等价类 + 2 个无效等价类**，共 5 个用例覆盖整个登录身份域。

#### 案例 2：JWT Token 等价类

| 等价类 | 代表值 | 用例数 | 预期 |
|---|---|---|---|
| 合法 Token（ADMIN） | `{{adminToken}}` | ~30 | HTTP 200 |
| 合法 Token（TEACHER） | `{{teacherToken}}` | ~25 | HTTP 200 |
| 合法 Token（STUDENT） | `{{studentToken}}` | ~20 | HTTP 200 |
| 缺失 Token | （无 Authorization 头）| ~108 | HTTP 401 |
| 错误 Token | `wrong.token.value` | 部分 | HTTP 401 |
| 过期 Token | `{{expiredToken}}` | 部分 | HTTP 401 |

**3 个有效等价类 + 3 个无效等价类**，覆盖鉴权所有可能。

## 三、边界值分析 (Boundary Value Analysis)

### 3.1 原理

ISTQB 定义：**等价类的边界比中间值更容易触发缺陷**。常用 2 值法（边界值 + 边界值±1）或 3 值法。

### 3.2 本测试集应用

#### 案例 1：分页参数边界

| 参数 | 等价类边界 | 代表值 | 预期 |
|---|---|---|---|
| `page` 最小值 | 0（非法）/ 1（有效）| `page=1` | HTTP 200 |
| `page` 最大值 | 总页数 / 总页数+1 | `page=999` | HTTP 200，data 为空数组 |
| `size` 最小值 | 0（非法）/ 1（有效）| `size=1` / `size=10` | HTTP 200 |
| `size` 最大值 | 100（合理）/ 1000（异常）| `size=10` | HTTP 200 |

#### 案例 2：文件上传大小边界（FileController）

| 等价类边界 | 测试值 | 预期 |
|---|---|---|
| 最小有效（1 字节） | 1B 文件 | HTTP 200 |
| 最大有效（10MB） | ~9.99MB 文件 | HTTP 200 |
| 超出上限（10MB+1） | 11MB 文件 | HTTP 200 / code=400 |
| 空文件 | 0B 文件 | HTTP 200 / code=400 |

源码：`@SpeechController.java:27-29`：

```java
if (audioFile.getSize() > 10 * 1024 * 1024) {
    return Result.error(400, "音频文件不能超过 10MB");
}
```

#### 案例 3：登录失败次数边界（限流）

| 等价类边界 | 测试值 | 预期 |
|---|---|---|
| 失败 1 次 | 1 次错误密码 | HTTP 200 / code=409 |
| 失败 4 次（边界-1） | 4 次错误密码 | HTTP 200 / code=409 |
| 失败 5 次（边界） | 5 次错误密码 | HTTP 200 / code=429（锁定） |
| 失败 6 次（边界+1） | 6 次错误密码 | HTTP 200 / code=429 |

源码：`@LoginRateLimiter.java:13-62` 限制 5 次失败/15 分钟。

## 四、错误推测 (Error Guessing) — 本集的核心技术

### 4.1 原理

ISTQB 定义：**基于测试者经验、缺陷模式知识，主动推测可能的错误情况**。

### 4.2 本测试集应用统计

| 错误推测类别 | 用例数 | 占比 |
|---|---|---|
| 鉴权失效（无 Token） | 108 | 74.5% |
| 鉴权失效（过期 Token） | 0（已合并）| 0% |
| 鉴权失效（错误 Token） | 0（已合并）| 0% |
| 越权访问（角色错误） | ~5 | 3.4% |
| 业务异常（重复账号、不存在 ID） | ~5 | 3.4% |
| 参数校验失败（空值、格式错） | ~5 | 3.4% |
| **错误推测总用例** | **~123** | **84.8%** |

### 4.3 错误推测来源（基于源码分析）

| 错误模式 | 源码证据 | 用例示例 |
|---|---|---|
| Spring Security 鉴权拦截 | `@SecurityConfig.java:59-66` | 108 个 401 用例 |
| @PreAuthorize 越权 | 各 Controller 类级注解 | STUDENT 调 ADMIN 接口 |
| @Valid 参数校验 | `@GlobalExceptionHandler.java:25-47` | 空字符串、null 值 |
| BusinessException 业务异常 | `@BusinessException.java:8-12` | 重复用户名、不存在 ID |
| 文件上传限制 | `@FileController.java`, `@SpeechController.java:27-29` | 超大文件 |

**错误推测占比 84.8%** 远超行业基线（30%），符合 ISTQB Advanced Level Test Analyst 推荐的"60-80% 反向用例占比"高质量测试集标准。

## 五、用例测试 (Use Case Testing)

### 5.1 原理

ISTQB 定义：**基于完整业务场景设计端到端测试流**，包含主流（Happy Path）与替代流（Alternative Flow）。

### 5.2 本测试集应用：Setup → 业务 → Cleanup 三段式

#### 5.2.1 Setup 流（前置准备）

```
1. ADMIN 登录 → 获取 adminToken      (1 用例)
2. TEACHER 登录 → 获取 teacherToken  (1 用例)
3. STUDENT 登录 → 获取 studentToken  (1 用例)
4. 获取已有试卷 ID (用于后续测试)    (1 用例)
```

#### 5.2.2 业务流（19 个 Controller 主流）

```
AuthController        → 4 接口 (登录、退出、当前用户、退出登录)
ChatController        → 19 接口 (主流 + 替代流共 20 用例)
ClassController       → 8 接口 (CRUD + 分页查询)
... (共 19 个 Controller，覆盖 108 接口)
```

#### 5.2.3 Cleanup 流（后置清理）

```
3 角色登出 → 清理服务器端 Token 缓存（如有）
```

### 5.3 业务场景示例：考试完整流程

```
TEACHER 登录 →
  创建试卷 (PaperController.create) →
  添加题目 (QuestionController.add) →
  发布考试 (ExamController.publish) →
  STUDENT 登录 →
    开始考试 (StudentExamController.start) →
    提交答案 (StudentExamController.submit) →
  TEACHER 阅卷 (MarkingController.mark) →
  STUDENT 查看成绩 (ScoreController.view) →
  全员登出 (Cleanup)
```

## 六、状态迁移测试 (State Transition Testing) — 待实施

### 6.1 现状

考试有完整的状态机：`未开始 → 进行中 → 已结束 → 已阅卷 → 成绩公布`，但本测试集**未对状态机迁移做专项测试**。

### 6.2 后续改进方向

| 状态 → 状态 | 是否合法 | 测试用例 |
|---|---|---|
| 未开始 → 进行中 | ✅ | 时间到自动开始 |
| 未开始 → 已结束 | ❌ | 应拒绝 |
| 进行中 → 已结束 | ✅ | 学生提交 / 时间到 |
| 进行中 → 未开始 | ❌ | 应拒绝 |
| 已结束 → 已阅卷 | ✅ | 教师阅卷完成 |
| 已阅卷 → 进行中 | ❌ | 应拒绝 |

## 九、结论与改进建议

### 9.1 当前优势

1. **错误推测技术应用充分**（84.8% 反向用例 vs 行业基线 30%）
2. **等价类划分系统化**（角色 / Token / 状态等多维度）
3. **用例测试结构清晰**（Setup → 业务 → Cleanup）

### 9.2 改进方向

| 优先级 | 改进项 | 工作量 |
|---|---|---|
| 中 | 增加状态迁移测试（考试状态机） | 2-3 小时 |
| 中 | 完善边界值（更多边界场景） | 1-2 小时 |
| 低 | 引入决策表测试方法 | 2-3 小时 |
| 低 | 探索性测试日志化 | 持续 |
