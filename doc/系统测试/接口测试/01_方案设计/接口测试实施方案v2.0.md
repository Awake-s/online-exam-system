> **作者**：陶展

> **质量目标**：上线零接口缺陷，符合 ISTQB Foundation Level + IEEE 829 + OWASP API Security Top 10 标准  
> **创建日期**：2026-04-30

## 0. v1.0 反思与 v2.0 改进承诺

### v1.0 的核心问题

| 问题 | 根因 | v2.0 改进 |
|---|---|---|
| 业务码假设 `code=0`（实际 `code=200`） | 套用阿里规范，未读 `Result.java` | **强制源码先行** — 每个断言贴源码引用 |
| 角色越权假设 HTTP 403（实际 HTTP 200+code=403） | 未读 `GlobalExceptionHandler.java` | **全异常处理映射表**作为方案附录 |
| 默认密码错误（数据库实际值 `admin123`/`123456`） | 未读 SQL 注释 | **接口画像**包含数据准备说明 |
| 反向用例污染 admin 触发 LoginRateLimiter | 未读 `LoginRateLimiter.java` | **测试隔离原则** — 反向用例用虚构账号 |
| 接口覆盖只 20/108 = **18.5%** | 范围定义过窄 | **分级覆盖策略**，P0+P1 ≥ 80% |

### v2.0 的工程化承诺

✅ **每个测试用例必须包含 4 个源码引用**：URL、Method、@PreAuthorize 注解、Result/Exception 行号  
✅ **每次提交前自动化追溯校验**：PowerShell 脚本扫描 Collection vs Controller，0 残差  
✅ **测试数据完全独立**：专用测试账号 `it_admin`/`it_teacher`/`it_student`，不污染业务数据  
✅ **测试用例双重审计**：自动校验（机器） + 人工 review（陶展自己）  
✅ **三方权威标准引用**：ISTQB / IEEE 829 / OWASP API Security Top 10

## 3. 测试用例设计方法论（每接口 6 类标准用例）

依据 **ISTQB Foundation Level Syllabus** 4.2 节「黑盒测试技术」，每个接口设计 6 类用例：

| # | 用例类别 | 设计依据 | 覆盖目标 |
|---|---|---|---|
| 1 | **正向用例** | 等价类划分（有效输入） | 接口功能正确性 |
| 2 | **参数缺失** | 边界值分析（缺失参数） | 参数校验 |
| 3 | **参数非法** | 边界值分析（越界、错类型） | 参数校验 |
| 4 | **业务异常** | 决策表测试（业务规则违反） | 业务逻辑 |
| 5 | **角色越权** | 状态转换测试（角色模型） | RBAC 鉴权 |
| 6 | **未登录访问** | 状态转换测试（认证模型） | JWT 鉴权 |

> 注：根据接口性质，并非每个接口都需要 6 类用例。例如：公开接口（如登录、验证码）不需要"未登录访问"用例。

### 用例数量预估

| 级别 | 接口数 | 平均用例数/接口 | 总用例数 |
|---|---|---|---|
| P0 | 41 | 4-5（含完整 6 类的子集） | 180 |
| P1 | 33 | 3-4 | 110 |
| P2 | 10 | 2-3 | 25 |
| **合计** | **84** | | **约 315 个用例** |

### 断言数量预估

每个用例平均 **3-5 个断言**（HTTP 状态码 + 业务码 + 业务字段 + 响应时间），  
**总断言数 ≈ 315 × 4 = 1260 个断言**。

## 4. 接口画像规范（强制 — 每个接口必填）

每个接口在编写测试用例**之前**，必须先输出一份"接口画像"，格式如下：

```yaml
# 示例：POST /api/auth/login
接口名称: 用户登录
URL: POST /api/auth/login
源码引用:
  - Controller: @AuthController.java:25 (@PostMapping("/login"))
  - Service: @AuthServiceImpl.java:69-110 (login)
  - DTO: @LoginRequest.java
鉴权要求: 公开（permitAll() · SecurityConfig.java:51）
请求参数:
  - username: String, 必填
  - password: String, 必填
  - captchaKey: String, 可选
  - captchaCode: String, 可选
返回结构:
  - 成功: Result<Map<String, Object>> {token, userInfo, roleCode}
  - 用户名/密码错误: BusinessException(409, "用户名或密码错误")
  - 限流锁定: BusinessException(409, "登录失败次数过多，请X分钟后再试")
  - 验证码错误: BusinessException(409, "验证码错误或已过期")
副作用:
  - 成功 → recordSuccess() 清除 LoginRateLimiter 计数
  - 失败 → recordFailure() 增加 LoginRateLimiter 计数（5 次锁 15 分钟）
测试用例:
  P0-Login-1: 正向 (admin/admin123) → code=200, data.token 非空
  P0-Login-2: 反向 (虚构账号) → code=409, message="用户名或密码错误"
  P0-Login-3: 反向 (缺 password) → code=400, message含"password"
```

## 5. 数据准备策略（测试隔离）

### 问题：当前用业务账号测试 → 污染数据 + 触发限流

### 方案：**专用测试账号 + 独立测试数据**

#### 5.1 测试账号（不与业务账号混用）

```sql
-- doc/接口测试/05_测试数据/init-test-users.sql
-- 创建专用测试账号（密码全部为 it123456）
INSERT INTO sys_user (id, username, password, real_name, role_id, status) VALUES
(9001, 'it_admin',   '$2a$10$...', '接口测试管理员', 1, 1),
(9002, 'it_teacher', '$2a$10$...', '接口测试教师',   1, 1),
(9003, 'it_student', '$2a$10$...', '接口测试学生',   1, 1);

-- 给 it_teacher 创建 1 张测试试卷
INSERT INTO exam_paper (id, paper_name, creator_id, ...) VALUES
(9001, '【接口测试】Java 测试卷', 9002, ...);

-- 给 it_teacher 创建 1 场测试考试
INSERT INTO exam_exam (id, exam_name, paper_id, teacher_id, ...) VALUES
(9001, '【接口测试】Java 期末测试', 9001, 9002, ...);
```

**好处**：
- ✅ 不污染真实业务数据
- ✅ 反向用例不污染真实账号的限流
- ✅ 测试运行后数据可一键清理（`DELETE WHERE id >= 9000`）

#### 5.2 LoginRateLimiter 处理

`LoginRateLimiter` 是基于内存的 ConcurrentHashMap，无持久化。处理策略：

```yaml
策略 A: 反向用例使用虚构用户名 nonexistent_test_xxx
  → 后端查询用户不存在 → 抛 BusinessException("用户名或密码错误") + recordFailure
  → 但因为是不同用户名，不影响真实账号
  → 风险：虚构用户名也会进入 LoginRateLimiter map（占用少量内存，可忽略）

策略 B: 接口测试前后通过测试数据 SQL 清理 LoginRateLimiter
  → 不可行：LoginRateLimiter 是内存对象，SQL 无法清理

策略 C: 接口测试前先调一次 captcha 接口（不影响限流）
  → 已采纳：作为 Setup-0 健康检查
```

## 7. 质量把关机制

### 机制 1：源码追溯强制性（每个用例必须有引用）

每个测试用例的 `event.test.script.exec` 第一行必须是源码引用注释：

```javascript
// SOURCE: @AuthController.java:25 + @AuthServiceImpl.java:69-110
// EXPECTED: code=200 (Result.java:17), data.token 非空
pm.test('HTTP 200', () => pm.response.to.have.status(200));
pm.test('业务码 200', () => pm.expect(json.code).to.equal(200));
// ...
```

### 机制 2：自动化追溯校验脚本

在 `03_PowerShell脚本/` 下增加 `audit-collection.ps1`：

```powershell
# 扫描 Collection，验证每个 Request 在后端 Controller 实际存在
# 不一致 → 输出错误 → exit 1
```

每次提交 Collection 前必须跑一次。

### 机制 3：双重审计

- **机器审计**：`audit-collection.ps1` 自动校验
- **人工审计**：陶展按 `06_源码追溯审计/接口源码追溯审计表.md` 逐项 review

### 机制 4：CI 集成（可选）

将 Newman 集成到 GitHub Actions / Jenkins，每次代码提交自动跑接口测试。
