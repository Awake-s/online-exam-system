# v8 课程描述统一升级说明 (2026-05)

## 一、背景

`edu_subject.description` 字段在 v6 升级后只接受文本,但实际入库的数据存在 **两套不同风格**:

| 来源 | 风格 | 字段数 | 例 |
|---|---|---|---|
| `06_科目表升级_年级专业课程化.sql` (14 条 id 100-113) | "类型 · 专业年级第N学期 · 学分/学时 · 内容" | **6 字段** | `专业核心必修 · 计科24级第5学期 · 4学分/72学时 · Servlet/JSP/Spring/MyBatis 等企业级开发` |
| `07_课程数据_22级与24级真实课表.sql` (1491 条 id 200+) | "类型·内容" | **2 字段** | `专业核心·SSM/Spring 框架` |

两套并存导致管理员视图里同一字段呈现不一致风格,影响答辩演示与培养方案对照阅读。

---

## 二、决策:**全部对齐为 6 字段格式**

```
[课程类型] · [专业简称][年级]第[N]学期 · [学分]学分/[学时]学时 · [核心内容]
```

例:`通识必修 · 通信22级第2学期 · 4学分/64学时 · 电路理论`

**选择 6 字段的 3 个理由**:

1. **信息自包含**:由于 v7 KISS 已物理删除 `course_type/credit/semester` 三列,这些维度的信息只能依附在 description 字段里
2. **答辩友好**:一行文字即可读完培养方案核心字段,不需多列展开
3. **向前兼容**:与已有的 06 升级 SQL 完全同构,不破坏现有 14 条记录的语义

---

## 六、覆盖范围

## 七、可重复执行

主生成器与 SQL 都遵循种子数据目录的 **可重复执行** 约定:
- 07 顶部 `DELETE WHERE id >= 100`,反复运行结果一致
- 06 用 `UPDATE WHERE id IN (100..113)`,反复运行结果一致
- 修改课程元数据后只需重跑 `python 工具_生成课程数据.py`
